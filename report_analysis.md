## أولاً: الملخص التنفيذي

**SystemMarket ERP** هو نظام متكامل لتخطيط موارد المؤسسات (ERP) مبني باستخدام Flutter/Dart وقاعدة بيانات SQLite المشفّرة via SQLCipher. يغطي النظام جميع العمليات المحاسبية والتشغيلية الأساسية: المشتريات، المبيعات، المخزون، نقاط البيع (POS)، المحاسبة المالية، الموارد البشرية، payroll، الأصول الثابتة، الميزانيات، والزكاة.

### الإحصائيات الرئيسية:
|المؤشر|القيمة|
|---|---|
|عدد الخدمات (Services)|~105|
|عدد جداول قاعدة البيانات|105 جدول|
|عدد وحدات النظام الرئيسية|7 وحدات|
|عدد مسارات UI|130+ مسار|
|عدد DAOs|32|
|طريقة التقييم|FIFO / AVCO / LIFO|
|التشفير|SQLCipher + FlutterSecureStorage|
|نظام القيد المحاسبي|Double-Entry (قيد مزدوج)|

### التقييم العام:
|الوحدة|التقييم (من 10)|ملاحظات|
|---|---|---|
|المشتريات|8.5|متكاملة مع GRN واعتماديات ومحاسبة تكاليف شحن |
|المبيعات|9.0|قوية مع عروض أسعار وأوامر بيع وفواتير إلكترونية (ZATCA)|
|المخزون|8.0|FIFO/AVCO/LIFO مع تتبع دفعات وصلاحية |
|نقطة البيع (POS)|8.5|متطورة مع ورديات وخصومات وأسعار جملة |
|التكامل المحاسبي|9.5|قيد مزدوج آلي بالكامل عبر PostingEngine |

## ثالثاً: تحليل دورة المشتريات

### 1. سير العمل (Workflow):

```
طلب شراء (اختياري)
    │
    ▼
أمر شراء للمورد ←── اعتماد (إذا تجاوز 10,000)
    │
    ▼
استلام البضاعة (GRN)
    │
    ▼
مطابقة GRN مع أمر الشراء والفاتورة
    │
    ▼
إنشاء فاتورة المورد ←── اعتماد متعدد المستويات (اختياري)
    │
    ▼
ترحيل الفاتورة (TransactionEngine.postPurchase)
    │
    ├──► تحديث المخزون (إنشاء دُفعات + تحديث الكميات)
    ├──► تخصيص تكاليف الشحن (Landed Cost)
    ├──► تحديث رصيد المورد (للمشتريات الآجلة)
    └──► ترحيل القيد المحاسبي (PostingEngine._postPurchase)
```

### 2. القيود المحاسبية للمشتريات:

**مشتريات نقدية:**
```
1040 المخزون (صافي قيمة البضاعة)      ... مدين
1050 ضريبة المدخلات (إن وجدت)         ... مدين
    1010 الصندوق                       ... دائن
```

**مشتريات آجلة:**
```
1040 المخزون (صافي قيمة البضاعة)      ... مدين
1050 ضريبة المدخلات (إن وجدت)         ... مدين
    2010 الذمم الدائنة (الموردون)       ... دائن
```

### 3. آلية تخصيص تكاليف الشحن (Landed Cost):

يقوم النظام بتوزيع تكاليف الشحن (landedCosts) على الأصناف بنسبة مئوية:
```
نسبة كل صنف = (قيمة الصنف / إجمالي قيمة الفاتورة)
التكلفة المخصصة = landedCosts × النسبة
تكلفة الوحدة النهائية = سعر الصنف + (التكلفة المخصصة / الكمية)
```

### 4. إنشاء الدُفعات (Batches):

عند ترحيل فاتورة المشتريات، يقوم النظام تلقائياً بإنشاء `ProductBatch` لكل صنف مع:
- رقم الدُفعة (PUR-XXX أو رقم وارد من المورد)
- تاريخ انتهاء الصلاحية
- تكلفة الوحدة (موزّع عليها تكاليف الشحن)
- الكمية المستلمة

### 5. نقاط القوة:

- ✓ التحقق من وجود GRN مرحّل قبل ترحيل الفاتورة
- ✓ اعتماد تلقائي للمشتريات التي تتجاوز 10,000
- ✓ إمكانية اعتماد متعدد المستويات (MultiLevelApprovalService)
- ✓ تخصيص تكاليف الشحن على مستوى الأصناف
- ✓ دعم تتبّع الدُفعات وتواريخ انتهاء الصلاحية
- ✓ التكامل التام مع القيود المحاسبية والمخزون

### 6. نقاط الضعف والمخاطر:

|الخطر|الوصف|التوصية|
|---|---|---|
|⚠️ لا يوجد فحص ضد تكرار رقم الفاتورة من نفس المورد|يمكن إدخال فاتورة مكررة لنفس المورد|إضافة تحقق من (supplierId, invoiceNumber) قبل إدخال الفاتورة|
|⚠️ لا يوجد تحقق من وحدات القياس عند الاستلام|قد يتم استلام كمية مختلفة عن أمر الشراء دون إنذار|إضافة حد تسامح (Tolerance) بين الكمية المطلوبة والمستلمة|
|⚠️ لا يوجد تدفق إجباري (Mandatory Workflow)|يمكن تخطي أمر الشراء والذهاب مباشرة للفاتورة|إعداد Policy إجباري: لا فاتورة بدون PO |
|⚠️ إمكانية استلام كميات أكبر من المطلوبة دون موافقة|GRN يقبل أي كمية|إضافة حد أعلى للاستلام (مثلاً 110% من الكمية المطلوبة)|

## رابعاً: تحليل دورة المبيعات

### 1. سير العمل (Workflow):

```
عرض سعر (Proforma Invoice) ─── اختياري
    │
    ▼
أمر بيع (Sales Order) ─── حجز المخزون (Reservation)
    │
    ▼
التحقق من المخزون (Stock Check) ←── تكسير التعبئة (Packaging Engine)
    │
    ▼
فاتورة مبيعات (Sales Invoice)
    │
    ▼
تسليم/شحن (Delivery Note) ─── اختياري
    │
    ▼
ترحيل الفاتورة (TransactionEngine.postSale)
    │
    ├──► خصم المخزون (FIFO/AVCO/LIFO)
    ├──► خصم الدُفعات (Batches with expiry)
    ├──► تحديث رصيد العميل (للمبيعات الآجلة)
    ├──► تحديث نقاط الولاء (Loyalty)
    ├──► تحديث عمولات المندوبين
    └──► ترحيل القيد المحاسبي (PostingEngine._postSale)
```

### 2. القيود المحاسبية للمبيعات:

**قيد الإيراد:**
```
1030 الذمم المدينة (للبيع الآجل)    ... مدين
1010 الصندوق (للبيع النقدي)         ... مدين
    4010 إيرادات المبيعات             ... دائن
    2020 ضريبة المخرجات               ... دائن
```

**قيد تكلفة البضاعة المباعة (COGS) - قيد منفصل:**
```
5010 تكلفة البضاعة المباعة           ... مدين
    1040 المخزون                       ... دائن
```

### 3. آلية احتساب التكلفة:

عند ترحيل فاتورة المبيعات:
- إذا وُجد `InventoryCostingService`: يستخدم طريقة التقييم المحددة للمنتج (FIFO/AVCO/LIFO)
- إذا لم يوجد: يستخدم FIFO كلوغاريتم احتياطي (أقدم دفعة أولاً)
- يتم خصم الكميات من الدُفعات بالترتيب: الأقرب لانتهاء الصلاحية أولاً (FEFO)

### 4. آلية تكسير التعبئة (Packaging Engine):

عندما تكون الكمية المطلوبة أقل من محتوى عبوة كاملة، يقوم النظام تلقائياً:
1. بفتح عبوة كاملة
2. إرجاع الكمية المتبقية كمخزون مفكوك (قطع سائبة)
3. تحديث all الدُفعات المرتبطة

### 5. نقاط القوة:

- ✓ دعم كامل لأنواع المبيعات: نقدي وآجل وتجزئة وجملة
- ✓ احتساب التكلفة بطرق متعددة (FIFO/AVCO/LIFO)
- ✓ FEFO لانتهاء الصلاحية
- ✓ حجز المخزون (Reservation) لأوامر البيع
- ✓ نقاط الولاء والعمولات
- ✓ QR code لمتطلبات ZATCA (السعودية)
- ✓ إصدار فواتير PDF مع QR

### 6. نقاط الضعف والمخاطر:

|الخطر|الوصف|التوصية|
|---|---|---|
|⚠️ يمكن بيع منتج بسعر أقل من التكلفة دون إنذار|لا يوجد تحقق من هامش الربح الأدنى|إضافة سياسة حد أدنى لهامش الربح|
|⚠️ الخصومات غير مقيدة بسقف على مستوى النظام|الحدود موجودة في الـ Permission فقط|إضافة حد أعلى للخصم كنسبة مئوية في Config|
|⚠️ لا يوجد مراجعة إلزامية للمبيعات الآجلة|كل المبيعات الآجلة ترحّل مباشرة|إضافة اعتماد إلزامي للمبيعات الآجلة فوق حد معين|

## خامساً: تحليل إدارة المخزون

### 1. طرق التقييم:

يدعم النظام ثلاث طرق لتقييم المخزون على مستوى **كل منتج على حدة**:
- **FIFO** (الوارد أولاً يصرف أولاً) — الافتراضي
- **AVCO** (متوسط التكلفة)
- **LIFO** (الوارد أخيراً يصرف أولاً)

يتم تحديد الطريقة عبر حقل `valuationMethod` في جدول `Products`.

### 2. آلية تحديث الكميات:

```
حركة شراء  ──► ProductBatches.quantity += كمية الدفعة
               Products.stock += كمية

حركة بيع    ──► ProductBatches.quantity -= كمية
               Products.stock -= كمية
               ربط بالتكلفة (FIFO/AVCO/LIFO)

تحويل مخزني ──► StockMovement: من مستودع → إلى مستودع
                StockTransfer + StockTransferItems

تسوية جرد   ──► StockTake + StockTakeItems
                InventoryAudit + InventoryAuditItems
```

### 3. معالجة التحويلات بين المستودعات:

عند تحويل مخزون بين مستودعين:
1. يتم إنشاء `StockTransfer` بسجل header
2. تسجيل `StockTransferItems` للأصناف المنقولة
3. إنشاء حركتي مخزون (StockMovement):
   - من المستودع المصدر: كمية سالبة
   - إلى المستودع الهدف: كمية موجبة
4. تحديث كميات الدُفعات في both المستودعين

### 4. الجرد الدوري والجرد المستمر:

النظام يستخدم نموذج **الجرد المستمر** (Perpetual Inventory):
- يتم تحديث المخزون فورياً مع كل عملية بيع أو شراء
- يدعم الجرد الدوري عبر `StockTake` للتسوية

### 5. معالجة المرتجعات والتالف:

مرتجعات المشتريات:
```
PurchaseReturn + PurchaseReturnItems
├──► StockMovement (إخراج من المخزون)
├──► تحديث رصيد المورد
└──► قيد محاسبي عكسي
```

مرتجعات المبيعات:
```
SalesReturn + SalesReturnItems  
├──► StockMovement (إدخال للمخزون)
├──► تحديث رصيد العميل
└──► قيد محاسبي عكسي
```

### 6. منع الرصيد السالب:

يقوم النظام بالتحقق من المخزون المتاح قبل كل عملية بيع:
```dart
if (sale.warehouseId != null) {
  final warehouseStock = await productsDao.getWarehouseStock(productId, warehouseId);
  if (warehouseStock < remainingToDeduct) {
    throw Exception('المخزون غير كافٍ');
  }
}
```

### 7. نقاط القوة:

- ✓ ثلاث طرق تقييم مع إمكانية اختيار الطريقة لكل منتج
- ✓ تتبع دُفعات مع تاريخ صلاحية
- ✓ تكسير التعبئة التلقائي
- ✓ منع الرصيد السالب على مستوى المستودع
- ✓ دعم serial numbers

### 8. نقاط الضعف:

|الخطر|التوصية|
|---|---|
|⚠️ لا يوجد حد أدنى للربح (selling price < cost مسموح)|إضافة Policy تحذيرية أو مانعة|
|⚠️ يمكن تغيير طريقة التقييم لمنتج بعد وجود حركات عليه|إضافة قفل لطريقة التقييم بعد أول حركة|
|⚠️ لا يوجد تحقق من تطابق وحدات القياس عند التحويل|إضافة تحويل تلقائي للوحدة الأساسية|

## سادساً: تحليل نظام نقطة البيع (POS)

### 1. آلية العمل (مبني على Bloc):

```
POSBloc (841 line)
├── Events (25 type)
│   ├── LoadCategories
│   ├── SelectCategory
│   ├── AddProductBySku / AddProductToCart
│   ├── UpdateQuantity
│   ├── ApplyDiscount
│   ├── CheckoutEvent
│   ├── ToggleReturnMode
│   ├── HoldCart / RecallCart
│   └── SwitchPriceList
│
├── State
│   ├── PosLoading

    ├── ربط الأصناف (SaleItems)
    ├── تطبيق الخصومات (على مستوى الفاتورة أو الصنف)

```

### 2. آلية فتح وإغلاق الكاشير (Shifts):

- **فتح الوردية**: يجب فتح وردية عمل (`Shift`) قبل أي عملية بيع نقدي
- **إغلاق الوردية**: يقارن الرصيد المتوقع (Expected Cash) مع الرصيد الفعلي (Closing Cash)
- **منع البيع النقدي بدون وردية**: حماية محاسبية

### 3. طرق الدفع:

- نقدي (Cash)
- آجل (Credit)
- بطاقة بنكية (Bank) — عبر تعريف PaymentMethod

### 4. المرتجعات والاستبدال:

- تتم عبر وضع إرجاع داخل POS (ToggleReturnMode)
- تُنشأ كفاتورة إرجاع (SalesReturn)
- القيد المحاسبي العكسي عبر PostingEngine

### 5. تأثير العمليات على:

|العملية|المخزون|الإيرادات|الصندوق|القيد المحاسبي|
|---|---|---|---|---|
|بيع نقدي|خصم كمية|زيادة إيرادات|زيادة نقدية|مدين: صندوق، دائن: إيراد + ضريبة|
|بيع آجل|خصم كمية|زيادة إيرادات|لا يتأثر|مدين: ذمم مدينة، دائن: إيراد + ضريبة|
|إرجاع بيع|إضافة كمية|خصم إيرادات|نقص نقدية/ذمم|قيد عكسي|
|خصم على الفاتورة|لا يتأثر|خصم الإيراد|لا يتأثر|صافي الإيراد = الإيراد - الخصم|

### 6. نقاط القوة:

- ✓ واجهة تفاعلية متطورة (Bloc مع 25 حدث)
- ✓ دعم الورديات مع إقفال إلزامي
- ✓ دعم الأوزان والأسعار المتعددة
- ✓ إرجاع واستبدال داخل POS
- ✓ تعليق واستدعاء الفاتورة (Hold/Recall)
- ✓ دعم الأرقام التسلسلية

### 7. نقاط الضعف:

|الخطر|التوصية|
|---|---|
|⚠️ لا يوجد تخزين مؤقت للفواتير في حال انقطاع الشبكة|إضافة Offline Queue للمبيعات|
|⚠️ الخصم على مستوى الصنف غير محدود بسقف (فقط الـ Permission)|إضافة Config لسقف الخصم المسموح به|
|⚠️ لا يوجد طباعة حرارية تلقائية بعد إتمام البيع|إضافة Auto-Print بعد Checkout|

## سابعاً: تحليل التكامل بين الوحدات

### 1. المشتريات ↔ المخزون:

```
PurchaseService.createPurchase()
  └─► inserts Purchase + PurchaseItems (Draft status)

TransactionEngine.postPurchase()
  ├─► التحقق من GRN موجود ومرحّل
  ├─► إنشاء ProductBatch لكل صنف مع التكلفة بعد توزيع الشحن
  ├─► تحديث Products.stock += الكمية
  ├─► InventoryTransaction (type: PURCHASE)
  └─► PostingEngine._postPurchase()
       └─► GLEntries + GLLines → AccountTransactions
```

### 2. المبيعات ↔ المخزون:

```
SalesService.createSale()
  └─► inserts Sale + SaleItems (Draft)

TransactionEngine.postSale()
  ├─► PackagingEngine.autoBreakIfNecessary() ← تكسير التعبئة
  ├─► InventoryCostingService.getBatchesForSale() ← استدعاء الدُفعات حسب الطريقة
  ├─► خصم الكميات من ProductBatches
  ├─► تحديث Products.stock -= الكمية
  ├─► InventoryTransaction (type: SALE)
  ├─► Mark serial numbers as SOLD
  └─► PostingEngine._postSale() → GLEntries (إيراد + COGS)
```

### 3. POS ↔ المحاسبة:

```
POS Bloc → CheckoutEvent
  └─► TransactionEngine.postSale()
       ├─► تحديث المخزون (مشترك مع المبيعات)
       ├─► التحقق من وردية مفتوحة للبيع النقدي
       └─► PostingEngine → GLEntries
```

### 4. المستودعات ↔ القيود المحاسبية:

كل مستودع (`Warehouse`) مرتبط بحساب GL (`accountId`):
```
Warehouse.accountId → GLAccounts.id (نوع فرعي من حساب المخزون 1040)
```

### 5. العملاء والموردون ↔ الأستاذ العام:

كل عميل ومورد يمكن ربطه بحساب GL:
```
Customer.accountId → Accounts Receivable (1030)
Supplier.accountId → Accounts Payable (2010)
```

## ثامناً: الفحص الرقابي واكتشاف المشكلات

### 1. العمليات غير المنطقية المكتشفة:

|#|الوصف|الخطر|
|---|---|---|
|1|إمكانية إنشاء فاتورة مبيعات بسعر بيع أقل من التكلفة|خسارة غير مكتشفة|
|2|لا يوجد تحقق من تكرار رقم فاتورة المورد|احتمال تكرار بنفس المبلغ|
|3|تغيير طريقة تقييم المخزون بعد وجود حركات|تذبذب في تقييم المخزون|
|4|إمكانية شراء بدون أمر شراء (PO غير إجباري)|ضعف الرقابة على المشتريات|

### 2. الثغرات المحاسبية:

|#|الوصف|التأثير|
|---|---|---|
|1|الفرق بين مدين ودائن في GLLines يتم التحقق منه ولكن لا توجد آلية auto-reversal|قد يعلق القيد في حالة الخطأ|
|2|PostingProfiles قابلة للتعديل وقد تؤدي لترحيل خاطئ إذا تم تغييرها|توزيع حسابات غير صحيح|
|3|حسابات العملاء (AR/AP) يتم تحديثها بشكل منفصل عن GLEntries (لا توجد علاقة FK مباشرة)|احتمال اختلاف الأرصدة|
|4|العملات المتعددة: يتم تخزين سعر الصرف في كل معاملة ولكن لا توجد إعادة تقييم للعملة في نهاية الفترة|فروقات عملة غير مسجلة|

### 3. المخاطر التشغيلية:

|الخطر|التوصية|
|---|---|
|قاعدة بيانات محلية SQLite مع تشفير — لا يوجد failover|إضافة نسخ احتياطي تلقائي واستعادة|
|جميع العمليات داخل transaction واحد — قد يسبب قفل للجدول مع عدد كبير من المستخدمين|تقييم أداء التزامن|
|لا يوجد سجل لـ Deleted Records (Soft Delete غير مستخدم)|إضافة Soft Delete للجداول الحساسة|
|البيانات الحساسة (passwords, encryption key) مشفرة ولكن قد تظهر في logs|مراجعة الـ debugPrint لإخفاء البيانات الحساسة|

### 4. احتمالات التلاعب:

|النقطة|التهديد|الحل المقترح|
|---|---|---|
|تعديل الفاتورة بعد الترحيل|يتم منع تعديل posted documents ولكن يمكن لـ Admin تجاوز|تدقيق إضافي على أي تعديل لحالة posted|
|حذف الأصناف من الفاتورة|الفاتورة المرحّلة لا يمكن حذفها ولكن الـ draft يمكن|تسجيل أي تعديل على draft في AuditLog|
|خصم غير مصرح به|التحقق من PermissionCode عند تطبيق الخصم|سقف خصم على مستوى Config مضاف لـ PostingEngine|

### 5. مشاكل الأداء المحتملة:

|المشكلة|السبب|
|---|---|
|حساب أرصدة الحسابات بشكل ديناميكي من GLLines كل مرة|لا يوجد caching لرصيد الحساب في جدول GLAccounts.balance|
|استعلامات batches داخل حلقة for لكل صنف|N+1 queries في دورة المشتريات والمبيعات|
|جميع العمليات في transaction واحد كبير|قد يسبب قفل قاعدة البيانات لفترة طويلة|

## تاسعاً: التوصيات الفنية والمحاسبية

### توصيات عاجلة (عالية الأولوية):

1. **إضافة كيان `VendorInvoiceNumber` فريد**: منع تكرار فواتير المورّدين بإضافة قيد UNIQUE على (supplierId, invoiceNumber).

2. **إضافة حد أعلى للخصم في Config**: بدلاً من الاعتماد فقط على نظام الصلاحيات.

3. **إضافة Soft Delete**: للجداول الحساسة (Purchases, Sales, GLEntries) مع سجل تدقيق.

4. **إضافة Revaluation للعملات الأجنبية**: معالجة فروقات العملة في نهاية الفترة.

### توصيات متوسطة الأولوية:

5. **إعادة هيكلة الـ caching لـ account balances**: إضافة caching لرصيد الحساب لتجنب حساب SUM في كل مرة.

6. **تفعيل الـ Approval Workflow الإجباري**: جعل أمر الشراء (PO) إلزامياً مع تدفق إجباري.

7. **إضافة Offline Queue لـ POS**: للعمل في حال انقطاع الشبكة.

8. **إضافة Periodic Auto-Recovery Check**: فحص دوري لتطابق أرصدة AR/AP مع GLEntries.

### توصيات طويلة المدى:

9. **RAID للبيانات المحلية**: نسخ احتياطي مشفّر تلقائي إلى cloud.

10. **إضافة Audit Trail لا يمكن تغييره**: استخدام AuditLogService مع توقيع رقمي لكل إدخال تدقيق.

11. **إضافة نظام تنبيهات (Alerts)**: تنبيه عند:
    - انخفاض المخزون عن حد إعادة الطلب
    - خصم يتجاوز النسبة المسموح بها
    - رصيد سالب متوقع
    - اقتراب انتهاء صلاحية الدفعات

12. **تحسين أداء الـ N+1 Queries**: استخدام batch loading لاستعلامات الدفعات في حلقات for.

## الملاحظات الختامية

**SystemMarket ERP** يعتبر نظاماً متطوراً يغطي معظم احتياجات المؤسسات المتوسطة. البنية التقنية (Clean Architecture, Drift, SQLCipher, Double-Entry) سليمة ومتينة. التنفيذ المحاسبي الدقيق عبر `PostingEngine` و `TransactionEngine` يضمن سلامة القيود المحاسبية. التحسينات المطلوبة تركّز على الجوانب الرقابية (ضمان عدم تكرار الفواتير، إجبارية الموافقات، حدود الخصم) وأداء الاستعلامات.

**الخلاصة**: النظام جاهز للاستخدام الإنتاجي مع التوصية بتطبيق التحسينات العاجلة (النقاط 1-4) لتعزيز الرقابة المالية.
